# Общий промпт для агентов: Engineering help по аргументам llama-server

Ты работаешь над подробной инженерной справкой llama-manager для аргументов
`llama-server`. Основной продукт находится в:

- `/home/maxim/llama/llama-manager`

Актуальный локальный checkout llama.cpp для сверки:

- `/home/maxim/llama/llama.cpp`
- commit смотри через `pnpm --filter @llama-manager/api args:docs:source-sync`
  или `git -C /home/maxim/llama/llama.cpp rev-parse HEAD`

## Задача

Для назначенных Markdown-файлов в `content/llama-args/llama-server` заменить
шаблонный черновик на подробную русскоязычную инженерную справку.

Каждый файл описывает один аргумент. Не объединяй аргументы в общий документ и
не переименовывай файлы без явной необходимости.

## Границы работы

- Редактируй только назначенные тебе файлы аргументов.
- Не меняй UI, API, тесты, lock-файлы, README и файлы других агентов.
- Ты не один в кодовой базе: другие агенты параллельно редактируют соседние
  документы. Не откатывай чужие изменения и не запускай форматирование всей
  папки.
- Если видишь, что назначенный файл уже существенно изменен другим агентом,
  аккуратно дополни его только в пределах своего аргумента.

Ссылки на GitHub issues/PR/discussions добавляй только когда источник реально
проверен или ссылка уже есть в справке/исходниках llama.cpp. Не добавляй
догадки вида "возможно обсуждалось".

## Что должно быть в хорошем документе

Сохраняй YAML frontmatter, но обновляй его по смыслу:

- `summary`: 1-2 практичных предложения, без шаблонных формулировок.
- `related`: оставляй только реально связанные аргументы.
- Не ломай `primaryName`, `aliases`, `category`, `valueType`, `valueHint`,
  `allowedValues`, `env`, если не нашел явную ошибку.
- Не добавляй `docStatus`, `reviewedHelpHash` и `reviewedLlamaCppCommit`:
  синхронизация теперь хранится в одном source snapshot.

Текст документа пиши на русском языке. Английские цитаты из `--help` можно
оставлять в блоке "Оригинальная справка llama.cpp".

Рекомендуемая структура:

- `# --argument`
- `## Кратко`
- `## Оригинальная справка llama.cpp`
- `## Паспорт аргумента`
- `## Что меняет в llama-server`
- `## Значения и формат`
- `## Когда использовать`
- `## Влияние на производительность и память`
- `## Взаимодействие с другими аргументами`
- `## INI-пресеты и router-режим` - если применимо
- `## Типовые проблемы и диагностика`
- `## Примеры`
- `## Источники`

Если раздел неприменим, напиши это коротко и конкретно. Не оставляй TODO и
автоматические фразы вроде "нужно проверить по исходному коду".

## Содержание, на которое стоит обращать внимание

- В какую структуру `common_params` записывается значение.
- На каком этапе применяется аргумент: парсинг CLI, загрузка модели, создание
  контекста, HTTP route, генерация, sampling, router subprocess.
- Принимает ли аргумент значения `on/off/auto`, отрицательные значения,
  специальные значения вроде `0`, `-1`, `all`, `none`.
- Есть ли парный `--no-*` аргумент и как он пишется в INI.
- Можно ли использовать аргумент внутри `--models-preset`.
- Какие аргументы управляются router-процессом и удаляются/перезаписываются
  при загрузке модели.
- Влияет ли параметр на RAM, VRAM, KV-cache, число слотов, latency, throughput,
  время старта или безопасность.
- Какие строки в логах помогают диагностировать проблему.
- Какие значения опасны для публичного сервера.

## Стиль

- Пиши как инженерную справку для администратора локального llama-server, а не
  как маркетинговое описание.
- Будь конкретен: "значение `0` отключает", "отрицательное значение запрещено",
  "путь проверяется как каталог на старте".
- Если есть неопределенность, явно укажи, что именно не подтверждено и каким
  источником это надо проверить.
- Не делай очень длинные общие вступления. Лучше больше практических деталей,
  примеров и диагностики.

## Проверка перед завершением

- YAML frontmatter остается валидным.
- В файле нет `TODO`, "создан автоматически", "нужно проверить" и других
  шаблонных остатков, кроме случая, когда неопределенность действительно
  описана как риск.
- Все упомянутые аргументы заключены в backticks.
- Примеры команд не требуют shell-конкатенации и показывают реальные формы
  аргумента.
- В финальном ответе перечисли измененные файлы и основные источники, которыми
  пользовался.
