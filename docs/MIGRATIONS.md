# Миграции конфигурации и состояния

Правило обратной совместимости: **мигрируем, не поддерживаем**. Любая ломающая
смена формата persisted-конфига или состояния на диске поставляется вместе со
стартовой миграцией, которая один раз переписывает старое в новое и **удаляет
источник**. Остальной код знает только текущий формат — это инвариант, который
делает миграции удаляемыми.

## Инвариант

Старый формат понимает ровно одно место — модуль миграции. Бизнес-логика и
Zod-схемы видят только актуальный формат. Поэтому убрать миграцию = удалить её
модуль + строку из `apps/api/src/migrations/registry.ts` + отметить здесь;
больше старый формат нигде не читается.

**Запрещено** для нашего persisted-конфига: read-time шимы, живущие вечно —
`z.preprocess(upgradeLegacy…)`, `?? legacyField`, `if (oldShape)` в схемах и
логике. Они и есть «мёртвый груз». Исключение — границы доверия (тело
HTTP-запроса от внешнего клиента), где мы осознанно принимаем чужой ввод; это
не миграция.

## Форма миграции

Один модуль на миграцию, объект `Migration` (`apps/api/src/migrations/types.ts`)
в упорядоченном массиве `registry.ts`. Драйвер `runMigrations()` идёт по
массиву, пропускает `isApplied()`, применяет остальные, возвращает применённые
id (логируются на старте). Требования:

- **идемпотентна и crash-safe**: пишем новое только если его нет, затем сносим
  старое; повторный запуск — no-op;
- **`isApplied()` определяет легаси структурно** (таблица существует / старый
  путь существует). Если формат структурно не отличить (смена формата
  значения), тогда заводим персистентный леджер применённых id — но не раньше,
  чем он впервые понадобится;
- **парсит легаси через Zod** → падает громко, не пишет мусор;
- **детерминированная запись** (стабильный порядок ключей) → чистые git-диффы
  в `data/config/`.

`id` — порядковый префикс `NNNN-slug`, монотонный, задаёт порядок выполнения.

## Удаление

Безопасно удалять, когда все реально работающие экземпляры рестартовали после
даты поставки миграции (после рестарта `isApplied()` уже `true` навсегда — на
диске нет легаси-состояния). Экземпляров немного, состояние боксов известно.

`proxy/legacy-migration.ts` пересоздаёт таблицу `api_proxy_runtime_metadata`,

`0005` заменила вечный read-time шим: `upgradeLegacyApiProxyPipeline`
(`packages/core/src/index.ts`) больше не висит в `z.preprocess` на
`ApiProxyPipelineConfigSchema`/`ApiProxyPipelineRecordSchema` — чтение
`pipelines.json` парсится строго новой формой. Функция осталась
экспортируемой, но вызывается явно только из миграций (`0002`, `0005`) и теста.
Когда `0005` можно удалить — вместе с ней удаляется и `upgradeLegacyApiProxyPipeline`,
и явный вызов в `0002`.
